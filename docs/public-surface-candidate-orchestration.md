# Public-surface candidate orchestration

Phase 3B-3 provides a local, fail-closed bridge between the source-owned
isolated generator and the website-owned immutable snapshot-retention layer.
It generates and retains review candidates only. It does not approve, adopt,
publish, deploy, confirm, or promote a candidate.

## Input contract

Run the command with every input explicit:

```text
node scripts/run-public-surface-candidate-orchestration.mjs \
  --source-repository <local-source-repository> \
  --SOURCE_COMMIT_APPROVED_FOR_GENERATION <40-lowercase-hex> \
  --generator-identity <same-exact-40-lowercase-hex-commit> \
  --website-repository <local-website-repository> \
  --python-executable <python-executable>
```

`SOURCE_COMMIT_APPROVED_FOR_GENERATION` is the only source-selection gate. It
is mandatory, exact, and external to both repositories. The command never
selects a branch tip, current `HEAD`, tag, release, timestamp-derived commit,
website-derived commit, or fallback commit. Missing, malformed, absent, or
mismatched input fails closed.

Phase 3B-1 carries generator identity through the immutable Git commit that
contains its builder, validator, and dependency-inventory schema; it adds no
generator field to candidate data or the dependency inventory. Phase 3B-3
therefore requires that commit identity explicitly and checks that it is the
exact commit archived and executed. For this phase the generator and source
material are independently extracted from the same approved commit.

## Structural flow

The command verifies the local source repository and exact commit, then uses
`git archive` to materialize separate temporary source and generator roots.
It never checks out, branches, commits to, or writes within the source
repository. Dirty working-tree bytes and source `HEAD` are not generation
inputs.

The generator sequence is fixed:

1. `validate_public_metadata.py --mode preflight`;
2. `build_public_surface_authority_map.py --source-root ... --output ...
   --inventory-output ...`;
3. exact output-set verification;
4. `validate_public_metadata.py --mode verify-inventory`;
5. fatal UTF-8 decoding, JSON parsing, and the existing website map contract;
6. SHA-256 over the exact generated bytes;
7. `retainPublicSurfaceSnapshot`, the existing Phase 3B-2 retention entry
   point.

Child processes receive argument arrays and never shell strings. Generation
runs outside both extracted roots with source-tree Python imports disabled.
Temporary state is beneath the website root only so Phase 3B-2 can receive a
safe repository-relative staging path; it is removed after success or failure.

## Retention and pointers

The website owns retained candidate bytes under its fixed immutable snapshot
root. The identity remains:

```text
<source-commit>-<exact-candidate-sha256>.json
```

Creation is exclusive and append-only. An identical existing snapshot is an
idempotent success; an identity collision with different bytes fails closed.
The orchestration layer calls the retention library instead of reproducing its
write logic.

`runtime-manifest.json`, `last-known-good.json`, and all pre-existing snapshot
bytes are audited before and after retention. The command has no active-pointer,
last-known-good, production-pointer, replacement, deletion, or adoption option.
Retention is not adoption.

## Deterministic result

Standard output is one LF-terminated JSON record with these mechanical fields:

```json
{"candidate_generation_record_schema_version":"public-surface-candidate-generation-record/1.0","source_repository":"metawritingecology/meta-writing-ecology","SOURCE_COMMIT_APPROVED_FOR_GENERATION":"<commit>","generator_identity":"<commit>","candidate_sha256":"<sha256>","retained_snapshot_path":"<repository-relative-path>","retention_result":"created","validation_result":"passed"}
```

`retention_result` is `created` or `identical-existing`. The record contains no
timestamp, user, host, machine path, temporary path, run ID, approval claim,
adoption claim, authority status, publication status, or production status.
Equivalent runs from equivalent inputs and initial retention state emit
byte-identical records.

## Failure model and phase boundary

Stable failures cover the approval gate, source repository and commit,
generator availability and identity, preflight/generation/inventory exits,
missing or ambiguous output, UTF-8, JSON, map contract, byte identity, unsafe
destination, existing-identity conflict, pointer mutation, source mutation, and
cleanup. No failure is downgraded to a warning followed by retention.

The command performs no network request and calls no GitHub API. It cannot
create a branch, commit, pull request, review, workflow, merge, deployment, or
production pointer. Phase 3B-4 owns later workflow preparation and remains
separate.

This layer is not an MWE registry, ontology engine, authority map, approval
mechanism, production-adoption mechanism, publication mechanism, or complete
representation of the MWE archive.
